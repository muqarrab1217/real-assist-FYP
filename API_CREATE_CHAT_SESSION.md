# API: CREATE CHAT SESSION
## Database Operation Documentation

---

## 📌 API Endpoint

**POST** `/api/chatbot/chat/create`

**Base URL:** `http://localhost:10000`

**Full URL:** `http://localhost:10000/api/chatbot/chat/create`

---

## 🔑 Authentication

**Required:** Yes (JWT Token)

**Header:**
```
Authorization: Bearer YOUR_JWT_TOKEN_HERE
```

The `user_id` is automatically extracted from the JWT token - you don't pass it in the request body.

---

### Headers

### Body Parameters (JSON)

| Parameter | Type | Required | Description | Example |
|-----------|------|----------|-------------|---------|

| `description` | string | No | Chat session description | "Discussing Lahore projects" |

```json
{}
```

**Recommended Request Body:**
```json
{
  "title": "Property Investment Questions",
  "description": "Discussing real estate options in Lahore"

```

## 🔄 What Happens in the Database

### SQL Operation (Behind the Scenes)

```sql
INSERT INTO chat_sessions (
  user_id,
  title,
  description,
  total_messages,
  is_summarized,
  status
) VALUES (
  'd218057f-13e3-4871-9ab6-0e612c30f0a1',  -- From JWT token
  'Property Investment Questions',           -- From request body
  'Discussing real estate options in Lahore', -- From request body or NULL
  0,                                         -- Auto-set to 0
  false,                                     -- Auto-set to false
  'active'                                   -- Auto-set to 'active'
)
RETURNING *;
```

### Database Fields Auto-Generated

| Field | Value | Source |
|-------|-------|--------|
| `id` | UUID | Auto-generated by database |
| `user_id` | UUID | Extracted from JWT token |
| `title` | string | Request body or "New Chat" default |
| `description` | string/null | Request body or NULL |
| `total_messages` | 0 | Hard-coded in API |
| `is_summarized` | false | Hard-coded in API |
| `status` | 'active' | Hard-coded in API |
| `created_at` | timestamp | Auto-generated by database |
| `updated_at` | timestamp | Auto-generated by database |
| `last_message_at` | null | NULL initially |
| `model_used` | null | NULL initially |

## 🔍 Behind the Scenes

### Step-by-Step Process

1. **Request arrives** at POST `/api/chatbot/chat/create`

2. **Authentication middleware** extracts JWT token:
   - Verifies token validity
   - Extracts `user_id` from token
   - Stores in `req.user`

3. **Request body** is parsed:
   - `title` → chatTitle (or "New Chat" default)
   - `description` → description (or null)

4. **Database insert** operation:
   ```javascript
   const { data: chat, error } = await db
     .from('chat_sessions')
     .insert([
       {
         user_id: userId,              // From JWT token
         title: chatTitle,             // From body or default
         description: description || null,  // From body or null
         total_messages: 0,            // Hard-coded
         is_summarized: false,         // Hard-coded
         status: 'active'              // Hard-coded
       }
     ])
     .select()
     .single();
   ```

5. **Response** is sent back with created chat object

## 📊 Database Schema Reference

### Table: `chat_sessions`

```sql
CREATE TABLE chat_sessions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES auth.users(id),
  title TEXT NOT NULL,
  description TEXT,
  total_messages INTEGER DEFAULT 0,
  is_summarized BOOLEAN DEFAULT false,
  status TEXT DEFAULT 'active',
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now(),
  last_message_at TIMESTAMPTZ,
  model_used TEXT
);
```

### Constraints
- `user_id` must exist in `auth.users` table
- `status` can be: 'active', 'archived', or 'deleted'
- `total_messages` is updated by triggers when messages are added

## 🛡️ Security Notes

1. **User Isolation:** Users can only create chats for themselves
   - The `user_id` comes from the JWT token (not request body)
   - Cannot create chats for other users

2. **Authentication Required:** All requests must include valid JWT token
   - Token is verified before processing
   - Expired tokens are rejected

3. **Input Validation:**
   - `title` is sanitized to prevent XSS
   - `description` is optional and can be null
   - Hard-coded values prevent malicious input

## 💡 Tips & Best Practices

1. **Save the Chat ID:** Always save the returned `chat.id` for future messages
2. **Use Descriptive Titles:** Helps users identify chats in history
3. **Add Descriptions:** Useful for context when reviewing old chats
4. **Token Expiry:** JWT tokens expire - refresh if you get 401 errors
5. **Error Handling:** Check for errors in response before using data
